# 信源画像

> **版本**: v1.2.0  
> **用途**: 记录已知解读信源的风格特征、偏向、盲区和使用历史，让 Skill 从"无状态工具"进化为"有记忆的系统"。  
> **本文档是模板**：包含方法论、评价体系和格式规范。具体的信源画像条目由**使用者**在使用过程中自行添加（见下方「使用者信源画像」章节）。  
> **粒度标准**: 信源画像以**作者/账号**为单位，不以平台为单位。同一平台的不同作者（如 CSDN 的不同博主）需创建独立画像条目。  
> **更新规则**: 每次完成六维度报告后，评估本次使用的信源，更新或新增画像条目。每次更新在条目末尾标注日期。

## 搜索通道说明

从 v1.1.0 起，Phase 1 搜索采用**分层 + 双通道**架构：

| 通道 | 覆盖范围 | 工具 | 信源类型 |
|------|---------|------|---------|
| **A: 通用搜索** | 全网（知乎、腾讯云、搜狐、CSDN、RSS 聚合、英文 blog 等） | `WebSearch` | 泛科技媒体、技术博客、个人专栏 |
| **B: 微信搜索** | 微信公众号文章（通过搜狗微信搜索接口） | `weixin_search` / `weixin_search_all` (MCP) | 公众号媒体、行业自媒体、学术公众号 |

### 分层通道策略

| 层级 | 通道 A 侧重 | 通道 B 侧重 |
|------|-----------|-----------|
| **Layer 1** | 全部策略关键词，中英双版本均发 | 全部策略，优先作者驱动 + 原文精确匹配 |
| **Layer 2** | L2_HIGHLIGHT（亮点猜测类泛化搜索） | L2_BIG_NAME + L2_TOOL_CHAIN（人物/工具锚点） |

### 差异化原因

微信公众号标题天然倾向 **「人物 IP + 工具/方法 + 成果亮点」** 三段式：
- `David Baker｜DNA binder 设计实现 100 倍跃迁`
- `Nature | David Baker团队用AI设计全新DNA结合蛋白`
- `RFdiffusion3 + AlphaFold3 | 从头设计DNA结合蛋白`

而 WebSearch 覆盖的泛科技媒体标题更多样，对中文概念泛化和长尾词更友好。

**不在这里给微信公众号做预画像。** 公众号是一个内容生态系统，每个公众号的画像将在 Phase 3 提取正文后按实际文章质量逐号评定，在 Phase 6 作为独立信源入库。这与「量子位」「机器之心」等站点的处理方式一致——先使用，后画像。

> **注意**：微信搜索 MCP 依赖 `fancyboi999/weixin_search_mcp`（PyPI: `weixin-search-mcp`），需在 `~/.workbuddy/mcp.json` 中配置。

## 使用指南

在 Skill 运行时，按以下优先级使用信源画像：

### Phase 0：加载画像

1. 读取本文件，获取所有已知信源的画像
2. 识别与当前论文领域/类型最匹配的信源及其特征（供 Phase 5 合成时参考）
3. **不限制 Phase 1 搜索范围**：画像仅作用于 Phase 5 合成阶段（信任权重分配），Phase 1 全源搜索不设域名过滤器

### Phase 5：批判性使用

根据画像特征调整对各信源信息的信任权重：

| 信息类型 | 优先信任的信源特征 | 需谨慎对待的信源特征 |
|----------|-------------------|---------------------|
| 技术细节 | 高深度、课题组/专业媒体 | 低深度、泛媒体 |
| 量化数据 | 高深度、有定量习惯 | 定性概括型、产业媒体 |
| 批判观点 | 高批判性、学术媒体 | 低批判性、官媒 |
| 产业判断 | 强中国视角、投资/产业媒体 | 纯学术媒体 |
| 宏观政策 | 官媒、强中国视角 | 纯技术媒体 |

### Phase 6：更新画像

报告完成后，评估本次使用的信源：
- 已有画像的信源：更新使用次数、最近使用日期，如果发现新特征则补充
- 首次出现的信源：在「使用者信源画像」章节末尾创建新画像条目

## 一、评价维度

信源画像的核心评价维度：

| 维度 | 评分 | 说明 |
|------|------|------|
| **技术深度** | ★☆☆☆☆ ~ ★★★★★ | 是否呈现方法细节、实验数据、定量结果？★越多越深入 |
| **批判性** | ★☆☆☆☆ ~ ★★★★★ | 是否对论文局限性、限制条件、未解决问题提出质疑？★越多越批判 |
| **乐观程度** | 非常乐观 / 乐观 / 审慎乐观 / 审慎 | 叙事基调偏突破叙事还是审慎评估？ |
| **中国视角** | 强 / 中 / 弱 / 无 | 是否从中国产业落地、政策、市场角度出发？ |

**评分参考**：

| 技术深度 ★ | 典型表现 |
|-----------|---------|
| ★★★★★ | 完整拆解方法管线，提供原始实验数据和精确数字 |
| ★★★★☆ | 覆盖主要方法环节，有具体数据但缺少细节 |
| ★★★☆☆ | 概述方法框架，定性描述为主，数据零星 |
| ★★☆☆☆ | 仅提及方法名称，无任何技术细节 |
| ★☆☆☆☆ | 纯新闻式报道，完全不涉及技术内容 |

| 批判性 ★ | 典型表现 |
|----------|---------|
| ★★★★★ | 系统性指出方法局限、数据偏差、未验证假设、转化距离 |
| ★★★★☆ | 讨论部分局限性，但叙事框架仍偏突破 |
| ★★★☆☆ | 偶尔提及个别限制，不深入展开 |
| ★★☆☆☆ | 仅结尾一笔带过"未来还需验证" |
| ★☆☆☆☆ | 完全不涉及任何局限性讨论 |

## 二、平台与粒度规则

| 平台类型 | 画像粒度 | 示例 |
|---------|---------|------|
| 机构媒体（澎湃新闻、科学网、科技日报） | 机构为单位 | "澎湃新闻"、"科技日报" |
| 品牌自媒体（如某领域专业公众号） | 品牌/公众号为单位 | "某某AI"（微信公众号） |
| 机构博客（高校/企业实验室官方博客） | 机构为单位 | "某某 Lab 官方博客" |
| 社区平台（CSDN、SegmentFault、知乎、腾讯云开发者社区） | **作者/账号**为单位 | "weixin_xxxxxx"（CSDN 某博主） |
| 自动摘要平台（如 gist.science） | 平台为单位 | "gist.science" |

**社区平台特别说明**：CSDN、SegmentFault、知乎、腾讯云开发者社区等是**社区**——不同作者/账号有截然不同的风格、技术深度和偏向。对于这些平台，画像条目必须以具体作者/账号为 key，不能笼统地按平台合并。同一个 CSDN 平台上的两个不同博主，应创建两条独立的画像条目。

## 三、画像条目模板

### 条目格式

在「使用者信源画像」章节中，按以下格式添加每个信源。复制下面的模板，填入实际信息：

```markdown
### {作者/账号名}（{平台名}）

- **作者/账号**: `{账号标识}`
- **平台**: {平台名}（{域名}）
- **URL 模式**: `{url_pattern}`
- **类型**: {机构媒体 / 品牌自媒体 / 个人博客 / 学术聚合 / 研究团队博客 / ...}
- **风格**: {一句话描述该信源的写作和叙事风格}
- **技术深度**: {★☆☆☆☆ ~ ★★★★★}
- **批判性**: {★☆☆☆☆ ~ ★★★★★}
- **乐观程度**: {非常乐观 / 乐观 / 审慎乐观 / 审慎}
- **中国视角**: {强 / 中 / 弱 / 无}
- **独特价值**: {该信源区别于其他信源的独特贡献——提供什么类型的信息是其他信源没有的}
- **盲区**: {该信源系统性地缺少什么内容——什么类型的信息它不覆盖}
- **首次记录**: {YYYY-MM-DD}（{首次使用的论文简称}）
- **最近使用**: {YYYY-MM-DD}
- **使用次数**: {N}
- **进化记录**: {如有值得记录的发现——如"本次提供了独特的横向对比数据""首次展现出某方面偏向"——在此追加。无新发现时省略。}
```

### 字段填写说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 作者/账号 | ✅ | 作者名、公众号名或账号 ID。社区平台必须使用实际账号 ID |
| 平台 | ✅ | 发布平台 + 域名，如"CSDN（blog.csdn.net）" |
| URL 模式 | ✅ | URL 通配模式，如 `blog.csdn.net/账号ID/article/details/*` |
| 类型 | ✅ | 从类型列表中选最接近的 |
| 风格 | ✅ | 一句话，如"学术技术流，侧重方法细节和架构分析" |
| 技术深度 / 批判性 | ✅ | 初始评分可保守（偏低），后续根据实际使用调整 |
| 乐观程度 | ✅ | 基于首次使用时的叙事基调判断 |
| 中国视角 | ✅ | 是否有中国产业/政策/市场视角 |
| 独特价值 | ✅ | 该信源不可替代的地方 |
| 盲区 | ✅ | 该信源系统性缺失的内容类型 |
| 首次记录 | ✅ | 首次使用的日期 + 论文简称 |
| 最近使用 | ✅ | 每次使用后更新 |
| 使用次数 | ✅ | 每次使用后 +1 |
| 进化记录 | 否 | 有值得记录的新发现时才添加 |

## 四、维护与更新规则

### 何时更新

每次完成六维度报告后（Phase 6），必须更新信源画像库：

1. **已有信源**：找到对应的画像条目，更新「最近使用」日期，使用次数 +1。如果本次使用中发现了新的特征（如"首次展现出产业判断能力"），追加到「进化记录」。
2. **新信源**：在「使用者信源画像」章节末尾，按「画像条目模板」格式创建新条目。

### 评分微调

- 首次评分可以保守（偏低）——低估比高估好修正
- 每次使用后如果发现评分与实际表现不符，在进化记录中标注并调整
- 技术深度和批判性的评分是基于该信源**多篇**文章的平均水平，不是单篇

### 归档策略

- 连续 10 次使用均未出现、且最近使用距今超过 6 个月的信源，可移入存档注释
- 不删除条目——在条目开头添加 `>[已归档] ` 标记即可

## 附录A：系统性偏向说明

基于对中文科技解读信源生态的持续观察，存在以下**系统性偏向**（非个别信源问题，而是整个信源生态的特征）：

| 偏向类型 | 表现 | 对报告的影响 |
|----------|------|-------------|
| **正面叙事偏好** | 报道成功案例，忽略失败案例 | 报告中可能遗漏论文中的无效结果和失败实验 |
| **量化松弛** | 用定性形容词替代精确数字 | 报告中可能缺失具体评分、精确数据 |
| **突破叙事** | 强调"首次""突破"，淡化渐进性 | 报告对论文实际贡献水平的判断可能偏乐观 |
| **技术细节跳过** | 不覆盖领域特定的纯技术推理环节 | 报告中可能缺失论文最精彩的推理链 |
| **竞品淡化** | 不报道或不强调竞品表现 | 报告对相对优势的判断可能偏乐观 |
| **自我警示忽略** | 不引用论文 Discussion 中的谦逊声明 | 读者可能不知道论文作者自己的谨慎态度 |
| **评估方法遗漏** | 不报道论文判断自身方法好坏的标准和指标体系 | 报告可能描述了方法功能，但缺失了"凭什么说这个方法好"的设计哲学 |

**重要说明**：此偏向是信源生态的固有特征，算法无法消除。太乙已将此偏向透明化处理（纳入报告免责声明），不尝试在报告中做算法性补偿。
