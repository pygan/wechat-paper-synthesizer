# 论文多源解读搜索与六维度信息合成系统

## 概述

本技能帮助无法获得文献原文的同学，通过检索全网多来源的文献解读，**多维度还原论文原文**，同时萃取专家观点，打造个人文献知识库。

**核心思路**：从不同解读信源中交叉验证论文原文信息，尽可能 100% 萃取原文核心内容。

**持续进化**：对信源的解读特点进行精准画像，持续提升萃取质量。

**六大维度严格二分**：忠实文献，又借鉴解读专家视角

| 类别      | 维度                  | 内容来源              |
| ------- | ------------------- | ----------------- |
| **文献的** | 研究目标、研究方法、实验验证、未来方向 | 论文原文（经由多信源交叉验证还原） |
| **解读的** | 批判分析、实用建议           | 解读文章对论文的点评与建议     |

## 报告结构：

1. **摘要**（论文摘要中译 + 信源核心观点）
2. **标签**（仅文献相关的关键词，5-8 个行内代码标签）
3. **文献的四个维度**（不归因于具体信源，多源交叉验证还原）
4. **解读的两个维度**（批判分析直接呈现问题、实用建议仅保留从业者启示）
5. **免责声明**（依据解读文章进行萃取，疏漏难免，建议查阅原文）
6. **附录**（论文基本信息含 PubMed 链接 + 信源列表含原文链接）
7. **OA 论文 PDF**（如为开放获取，自动下载至报告同目录）

## 核心价值：

1. **萃取还原**：从解读文章的二手信息中交叉验证萃取原文，尽可能 100% 还原
2. **搜索管道**：分层递进搜索（Layer 1 标准覆盖 → Layer 2 模块化升级）+ 双通道（WebSearch + 微信公众号）+ WebFetch 提取
3. **文献/解读严格分离**：读者一目了然什么是论文原文内容、什么是专家解读
4. **去 AI 味**：批判分析不回避问题，实用建议具体到可操作层面
5. **本地归档**：Markdown + PDF 双格式，可与 Obsidian 等工具无缝衔接

## 完整工作流程

```
用户输入
  ├─ 提供论文截图 → 先用 Read 工具识别论文元数据
  ├─ 提供论文标题/DOI → 直接进入搜索阶段
  └─ 提供非论文关键词 → 澄清：需要具体论文信息才能搜索
            ↓
确定论文元数据（DOI、标题、作者、期刊、年份、类型）
            ↓
[Phase 0] 加载信源画像库，匹配已知信源
            ↓
[Phase 1] 分层搜索
  ├─ Layer 1: 标准搜索（四策略 + 中英混合 + 双通道并行）
  ├─ 阈值判断：≥3 篇不同信源？
  │    ├─ YES → Phase 2
  │    └─ NO  → Layer 2: 模块化升级（大牛锚点/亮点猜测/工具方法链）
  └─ 合并结果
            ↓
[Phase 2] 标题筛选：识别直接相关的解读文章
            ↓
[Phase 3] WebSearch 反查真实URL + WebFetch 提取正文
            ↓
[Phase 4] （可选）下载 OA 论文 PDF
            ↓
[Phase 5] 六维度合成报告（基于信源风格批判性使用）
            ↓
[Phase 6] 更新信源画像库
            ↓
         交付报告 + PDF
```

## 关键步骤说明

### Phase 0: 加载信源画像库

#### 目的

加载已知信源的特征摘要，**供 Phase 5 合成时使用（信任权重分配、盲区补偿）**，加载画像不限制搜索阶段搜索的范围。

#### 执行步骤

1. 读取 `references/source_profiles.md`
2. 识别与当前论文领域/类型最匹配的已知信源及其特征

#### 重要边界

- **Phase 1 搜索不受画像限制**：搜狗+WebSearch 全源搜索，不设域名过滤器，已知信源和新信源在搜索结果中地位平等
- **画像仅作用于 Phase 5 合成**：根据画像评分决定各类信息从哪个信源优先采用，不用于过滤搜索结果
- **新信源在 Phase 6 入库**：每次搜索发现的新信源，在 Phase 6 创建画像条目，下次即可被识别

#### 输出

- 已知信源的匹配列表和特征摘要（供 Phase 5 参考）

### Phase 1: 搜索阶段（分层架构）

搜索分两层递进：Layer 1 快速覆盖 → 阈值判断 → Layer 2 深度补漏。

```
Layer 1: 标准搜索（四策略 + 中英混合）
  ├─ 双通道并行
  └─ 阈值判断：≥3 篇不同信源完整正文可提取？
       ├─ YES → 直通 Phase 2
       └─ NO  → 触发 Layer 2

Layer 2: 升级搜索（模块化策略，条件触发）
  ├─ 根据论文特征自动激活策略模块
  ├─ 通道差异化执行（A侧重泛化，B侧重人物/工具锚）
  └─ 合并 Layer 1 + Layer 2 → Phase 2
```

#### 1.1 提取论文元数据

从用户输入中提取以下字段，供后续关键词构造使用：

| 字段 | 用途 | 示例 |
|------|------|------|
| 通讯作者/大牛 | L1 作者驱动、L2 大牛锚点 | David Baker |
| 论文标题英文关键词 | L1 原文精确匹配 | DNA binder, de novo design |
| 期刊名 + 年份 | L1 原文精确匹配 | bioRxiv, 2026 |
| 核心方法/工具名 | L2 工具方法链 | RFdiffusion3, AlphaFold3 |
| 摘要中的关键数字/声明 | L2 亮点猜测 | 100-fold, first demonstration |
| 中文核心概念 | L1 中文泛化 | DNA结合蛋白、蛋白质从头设计 |

#### 1.2 Layer 1: 标准搜索

##### 1.2.1 构造 Layer 1 关键词（四策略 + 中英混合增强）

在原有四策略基础上，每组关键词**必须生成中英双版本**（中英混合版 + 纯中文版）。详细规则见 `references/search_queries.md`。

| 策略 | 数量 | 中英混合版示例 | 纯中文版示例 |
|------|------|-------------|------------|
| 原文精确匹配 | 2-3 组 | `"Nature" "DNA binder" 2026 蛋白设计 解读` | `《自然》 DNA结合蛋白 从头设计 2026 解读` |
| 作者驱动 | 1-2 组 | `"David Baker" "DNA binder" "de novo" 蛋白质设计` | `David Baker DNA结合 从头设计 蛋白质` |
| 中文泛化 | 1 组 | `DNA binder 从头设计 蛋白质 人工智能 特异性` | `DNA结合蛋白 从头设计 人工智能 特异性识别` |
| 长尾试探 | 0-1 组 | `"可编程基因调控" "de novo" protein binder` | `可编程基因调控 蛋白设计 基因编辑` |

**中英混合规则**：
- 英文专业术语（binder、diffusion、design、screening）**保留英文**——中文科技写作者常用中英混杂
- 期刊名优先英文（`"Nature"`），同时生成中文版（`《自然》`）
- 方法/工具名保留英文原词（`RFdiffusion`、`AlphaFold`）
- 每个策略生成的「中英混合版」和「纯中文版」各为一组独立搜索词

##### 1.2.2 双通道并行执行

使用 Layer 1 所有关键词在两个通道**一次性并行**发出：

| 通道 | 工具 | 覆盖范围 | Layer 1 关键词侧重 |
|------|------|---------|------------------|
| **A: 通用搜索** | `WebSearch` | 全网（知乎、腾讯云、搜狐、CSDN、RSS 聚合、英文 blog 等） | 全部策略关键词，中英双版本均发 |
| **B: 微信搜索** | `weixin_search` / `weixin_search_all` (MCP) | 微信公众号文章（搜狗微信搜索接口） | 全部策略关键词，**优先作者驱动 + 原文精确匹配**（公众号标题对人物和期刊锚点更敏感） |

- 微信搜索每组 max_pages=2
- 所有搜索词在 Layer 1 一次性并行发出，不等待结果再补搜

##### 1.2.3 阈值判断

Layer 1 搜索完成后，合并去重、相关度初筛。判断是否满足**充分性阈值**：

| 判定 | 条件 | 动作 |
|------|------|------|
| **充分** | ≥ **3** 篇来自 ≥ **2** 个不同平台/公众号的完整正文可提取 | 直通 Phase 2 |
| **不足** | < 3 篇可提取，或所有可提取正文来自同一平台/公众号 | 触发 Layer 2 |

**平台区分规则**：「蓝极BlueArctic」（微信公众号）、知乎专栏、搜狐号、腾讯云开发者社区各算不同平台。同一公众号的多篇文章算同一平台。

#### 1.3 Layer 2: 升级搜索（条件触发）

仅当 Layer 1 未达阈值时激活。

##### 1.3.1 触发条件

满足以下**任一条件**即触发 Layer 2：
- 可提取完整正文的信源 < 3 篇
- 所有可提取信源来自同一平台
- 噪音率 > 70% 且标题过滤后仍无可匹配文章

##### 1.3.2 策略模块一览

Layer 2 策略采用**模块化设计**——每个策略是独立模块，可单独激活/停用，新增策略只需添加模块定义，不改变搜索框架。详见 `references/search_queries.md`。

| 模块 ID | 策略名称 | 触发条件 | 通道偏好 | 核心思路 |
|---------|---------|---------|---------|---------|
| `L2_BIG_NAME` | 大牛锚点 | 通讯作者为领域知名人物 | **B > A** | `"{大牛名}" + {领域词}`，不绑定期刊/年份 |
| `L2_HIGHLIGHT` | 亮点猜测 | 摘要含显著数字声明 | A > B | 从数字/声明反向构造"成果包装"式搜索词 |
| `L2_TOOL_CHAIN` | 工具方法链 | 论文涉及 ≥2 个知名计算工具 | **B > A** | 用工具名组合搜索 |

**模块扩展接口**：新增策略模块只需在 `search_queries.md` 的 `## Layer 2 策略模块定义` 章节中添加一个新模块条目（含触发条件、通道偏好、关键词模板、参数来源）。无需修改 SKILL.md。

##### 1.3.3 通道差异化执行

Layer 2 各通道使用**不同关键词**，根据模块的通道偏好分配：

| | 通道 A: WebSearch | 通道 B: 微信搜索 |
|---|---|---|
| **主要承担** | `L2_HIGHLIGHT`（亮点猜测类泛化搜索） | `L2_BIG_NAME`、`L2_TOOL_CHAIN`（人物/工具锚点） |
| **关键词风格** | 偏中文概念泛化 + 成果声明 | 人物锚点 + 英文工具名 + 成果数字 |
| **示例** | `"蛋白质从头设计 100倍 特异性 DNA 2026"` | `"David Baker" RFdiffusion3 DNA binder 100倍` |

**为什么差异化？** 微信公众号标题天然倾向于「人物 IP + 工具/方法名 + 成果亮点」的格式（如 `David Baker｜DNA binder 设计实现 100 倍跃迁`），而 WebSearch 覆盖的泛科技媒体对概念泛化和中文长尾词更友好。

##### 1.3.4 Layer 1 + Layer 2 结果合并

- Layer 1 和 Layer 2 结果跨层去重
- Layer 2 补充的候选文章标注来源层级 `[L2]`
- 合并后进入 Phase 2 筛选

#### 1.4 噪音率预估

无论 Layer 1 或 Layer 2，搜索结果合并后评估噪音率。噪音率预估表见 `references/search_queries.md`。若噪音率 > 70%，在报告中标注。

#### 1.5 前置条件（MCP 配置）

通道 B 依赖 `weixin_search_mcp` MCP 服务器。如未配置，通道 B 自动降级为仅使用 WebSearch（Layer 2 中 B 通道的搜索词改用 A 通道执行）。

### Phase 2: 筛选阶段

从搜索结果中识别「直接解读目标论文」的文章。判断标准：

1. **必含要素**：标题同时出现论文的作者名 (或姓氏) + 期刊名 + 核心关键词
2. **排除条件**：
   - 仅提及作者但不涉及目标论文
   - 仅提及期刊但内容不相关
   - 时间明显不符（如论文 2026 年发表但文章发布于 2024 年前）

输出一个精选列表（建议 3-8 篇），标注每篇的匹配度。

---

### Phase 3: 提取阶段

#### 3.1 URL 反向查找

对于搜索引擎返回的每篇候选文章标题，使用 `WebSearch` 工具通过完整标题搜索真实 URL。

#### 3.2 正文提取

使用 `WebFetch` 工具提取文章全文。

平台兼容性矩阵：

| 平台           | 可提取 | 备注      |
| ------------ | --- | ------- |
| sohu.com     | ✅   | 直接提取    |
| thepaper.cn  | ✅   | 直接提取    |
| wxredian.com | ✅   | 转载站，可提取 |
| news.qq.com  | ✅   | 直接提取    |

**提取质量标准**：

- 至少获取 2 篇不同信源的完整正文（如无法达到，标注信息缺口）
- 优先选择专业科学媒体（如"生物世界""知识分子""返朴""深究科学"）和课题组官方账号
- 避免依赖单一信源

#### 3.3 内容质量速评

对每篇提取成功的文章，记录：

- 作者/公众号属性（行业媒体 / 课题组 / 投资人 / 泛媒体）
- 技术深度（定性 / 定量 / 有图 / 有原始数据）
- 是否有批判性内容或仅纯赞美/翻译

### Phase 4: 下载 OA 论文 PDF（可选）

如果论文是开放获取（Open Access），下载 PDF 原文。

- 通过 DOI 构造链接：`https://doi.org/{DOI}` → 解析到出版商页面 → 查找 PDF 链接
- 或直接通过 PubMed Central（PMC）获取：`https://www.ncbi.nlm.nih.gov/pmc/articles/{PMCID}/pdf/`
- 使用 curl/wget 下载，或通过 WebFetch 获取后写入文件

文件命名规则：

```
paper_{第一作者姓氏}_{期刊}_{论文题目核心词}_{核心关键词}.pdf
```

保存到报告同目录。

如论文非 OA，跳过此步骤，在报告中标注 OA 状态。

### Phase 5: 合成阶段

#### 5.1 报告结构

严格按照 `assets/report_template.md` 的结构生成报告：

```
摘要（论文摘要中译 + 信源核心观点三个一句话概括）
  ↓
标签（文献关键词 5-8 个，行内代码样式）
  ↓
一、研究目标 [文献的]
二、研究方法 [文献的]
三、实验验证 [文献的]
四、未来方向 [文献的]
  ↓
五、批判分析 [解读的]（直接呈现问题：先呼应摘要三点+论述，再补充其余）
六、实用建议 [解读的]（仅保留从业者启示表格）
  ↓
免责声明
  ↓
附录 A：论文基本信息（含 PubMed 链接）
附录 B：解读文章信源（含原文链接）
```

#### 5.2 文献的四个维度

详细指南见 `references/six_dimensions.md`。核心要点：

| 维度   | 核心任务                | 禁止事项                |
| ---- | ------------------- | ------------------- |
| 研究目标 | 还原论文定位 + 假设 + 背景    | 加入解读文章的主观评价         |
| 研究方法 | 还原完整技术管线 + 组件表      | 堆砌术语不加解释            |
| 实验验证 | 还原关键数字 + 验证设计 + 里程碑 | 数字无出处 / 加入解读的"意味"分析 |
| 未来方向 | 还原论文作者提出的未来方向       | 混淆作者说的与解读猜的         |

**交叉验证规则**：

- 同一事实出现在 2+ 信源 → 直接呈现
- 仅出现在 1 个信源 → 标注「单信源」
- 多信源数字不一致 → 标注差异，取最保守值

#### 5.3 解读的两个维度（基于信源风格批判性使用）

| 维度   | 核心任务                                         | 禁止事项               |
| ---- | -------------------------------------------- | ------------------ |
| 批判分析 | 直接呈现需要客观看待的问题（先呼应摘要三点，再补充其余），汇总解读文章对论文的分析与质疑 | 对解读文章本身做批判 / 设铺陈段落 |
| 实用建议 | 从解读中提炼可操作洞察（仅从业者启示表格）                        | 说"值得关注"而非"关注什么信号"  |

**基于信源风格的信息使用规则**：

| 信息类型      | 优先采用的来源特征                   | 需要交叉验证的来源特征     |
| --------- | --------------------------- | --------------- |
| 技术细节、实验数据 | 高深度、专业媒体（DrugAI、生物世界）       | 低深度、泛媒体、官媒      |
| 量化评分、精确数字 | 高深度、有定量习惯（DrugAI、ScienceAI） | 定性概括型、产业媒体（智药局） |
| 批判性质疑     | 高批判性、学术媒体（DrugAI、ScienceAI） | 低批判性、官媒（科技日报）   |
| 产业判断、商业化  | 强中国视角、产业媒体（智药局）             | 纯学术媒体           |
| 政策视角      | 官媒、强中国视角（科技日报）              | 纯技术媒体           |

批判分析中，观点来自解读文章则标注信源，观点为 AI 独立发现则在末尾标注「本报告补充」。不设"选题建议""进一步核实清单"等扩展内容。

**信源单一时的补充策略**：如果仅从 1-2 个信源获取信息，基于信源画像主动补充该信源盲区中可能遗漏的问题类型（如产业媒体缺失技术细节、专业媒体缺失产业判断），并在报告中标注「基于信源画像推断」。

#### 5.4 报告输出

生成的 Markdown 报告保存到工作目录，文件命名规则：

```
paper_analysis_{第一作者姓氏}_{期刊}_{论文题目核心词}_{YYYYMMDD}.md
```

PDF 文件命名规则（如有）：


### Phase 6: 更新信源画像库

#### 目的

每次使用都是对 Skill 知识库的贡献——信源画像越准确，后续使用的效果越好。

#### 执行步骤

1. 评估本次使用的每个信源：

   - 画像中的评分是否需要微调？

## 局限性声明

1. **噪音率控制**：综述类/热门话题的噪音率超过80%，当前仅用标题关键词过滤，未使用语义匹配。需要后续版本引入更强的相关度评分。
2. **单信源风险**：如果仅找到 1 篇解读，批判分析维度和交叉验证均减弱。此时在报告中明确标注，并基于信源画像补充该信源盲区中可能遗漏的问题。
3. **语言限制**：当前搜索和合成以中文内容为主。英文解读（Medium/Twitter/Substack）暂未纳入覆盖范围，后续版本可扩展。
4. **OA PDF 下载**：仅支持明确标注 Open Access 的论文。付费墙论文（如 Nature 非 OA 文章）无法自动下载。
5. **信源画像冷启动**：新安装的 Skill 或新领域论文可能面临信源画像库不足的问题。此阶段应积极创建新画像条目，积累 3-5 次使用后画像库将趋于实用。
6. **科技解读系统性偏向**：信源生态本身存在正面叙事偏好、量化松弛、失败案例缺失等系统性偏向（详见 `references/source_profiles.md` 系统性偏向说明）。此为信源固有特征，Skill 无法消除，仅在免责声明和批判分析中做透明化处理。
